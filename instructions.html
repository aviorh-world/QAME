🗺️ The Big Picture

You record a flow once by clicking through it like a normal user. The bookmarklet captures every URL, every CTA you clicked, every value you typed. Drop the resulting bundle into the tool, pick "Generate replay", and you get a Playwright .spec.ts file that walks through the same flow every time.

🚀 Quick Start (5 Minutes)

Step 1 — Install the Bookmarklet

Open the tool's HTML page in your browser. In section 1, drag the green 🤖 Auto Flow Runner button to your browser's bookmarks bar.

Tip: If your bookmarks bar isn't visible, press Ctrl+Shift+B (Windows) or Cmd+Shift+B (Mac).

Step 2 — Record a Flow

  1. Navigate to the first page of your flow (e.g. a checkout start page).
  2. Click the 🤖 Auto Flow Runner bookmark.
  3. A floating panel appears in the bottom-right corner.
  4. Click ▶ Play in the panel to start automatic playback, OR click Step to walk through manually one CTA at a time.
  5. Type any values the form needs (the bookmarklet will learn them).
  6. When done, click 📦 Bundle (or Export) in the panel to download a qa-bundle-*.json file.

Step 3 — Drop the Bundle

Back in the tool, in section 2, drop the qa-bundle-*.json file onto the card (or paste its contents). The results panel appears immediately showing stats and a table of everything recorded.

Step 4 — Generate the Replay

Expand section 4 — Generate replay and choose one of:

  • 🎧 Download Playwright spec (.ts) — for running in CI / on a server
  • 📋 Download console replay (.js) — for pasting into DevTools to run live in your browser

Files are auto-named based on the site (e.g. flow-replay-mysite.spec.ts).

Step 5 — Run It

In your terminal, from your Playwright project root:

npx playwright test --ui

This opens the Playwright UI mode — the recommended way to debug. You'll see the spec, can run it step-by-step, watch screenshots between actions, see network calls, and time-travel through any moment of the test.

Links to the demo video and this README, plus three quick reminders.

The drag-to-bookmarks-bar button. One-time install.

Section 2 — Drop the bundle

Drop zone for the qa-bundle-*.json you exported from a recording. Updates the results panel live.

Section 3 — Start URL or API (optional)

Static URL: Force the spec to always start at this URL.
API endpoint: Have the spec POST/GET to an API at runtime and use the returned URL. Useful when each test run needs a fresh token.

Inside this section:

  • JSON path — Where to find the URL in the API response. Leave empty to auto-detect common field names (url, RedirectURL, applicationUrl, etc).
  • Custom headers — For API auth.
  • Request body — For POST.
  • Learned Rules — Per-domain field overrides as JSON. Used when the bundle's auto-learned values aren't right.

Section 4 — Generate replay (collapsed by default)

Two buttons:

  • Playwright spec (.ts) — Full-featured test file.
  • Console replay (.js) — Standalone JS to paste in browser DevTools.

💡 Things Worth Knowing

1Run stopped mid-way?
The bookmarklet stores everything in sessionStorage. Click the bookmark again on the page you got stuck on — it resumes from exactly that point. No need to start over.
2Can't auto-fill a field?
This happens with iframes, custom date-pickers, and any non-standard input. Just type the value yourself. If Learn mode is on, the bookmarklet captures what you typed, and the generated Playwright spec will know to fill it the same way on every run.
3Page looks stuck but nothing's wrong?
Check the Log button in the panel. The bookmarklet might be waiting on a cookie banner it doesn't recognize, or a CTA whose text changed. The console-replay generates the same logs in your browser console with color-coded output.
4Generated spec runs in Playwright but step X fails?
Check the flow-replay-results.json attachment in the Playwright HTML report. It tells you exactly which step failed and why (URL mismatch, CTA not found, etc.). Most failures are because production changed pages — re-record the flow.
5The console-replay vs the Playwright spec
Console replay: For quick smoke tests in your own browser. No CI setup. Paste in DevTools console, watch it run.
Playwright spec: For CI, parallel runs, traces, screenshots, network mocking. Real test infrastructure.
Both share the same logic — dedup, skip-ahead, auto-advance, generic CTA fallback, cookie dismissal. Use whichever fits.

⚙️ Smart Logic — How the Replay Handles Real-World Mess

Production isn't a clean recording. The replay code handles common issues automatically:

ProblemWhat the replay does
Recording captured 6 duplicate clicks on the same buttonDedup: collapses consecutive identical (URL, CTA) entries into one
Production shows an extra page that wasn't in the recordingAuto-advance: fills it and clicks a generic CTA up to 3 times
Production skipped a page that was in the recordingSkip-ahead: detects we're on a later step's URL and jumps to it
Cookie banner blocks the clickCookie dismissal: tries 50+ selectors for major CMPs (OneTrust, Cookiebot, Didomi, Quantcast, …) plus a DOM-scan fallback
Recorded CTA name doesn't match the actual buttonGeneric CTA fallback: tries any common CTA word (Continuar, Siguiente, Confirmar, Submit, …)

After 2 consecutive failures the replay bails out cleanly with a SKIPPED status, so it never spins forever.

🛠️ Running the Playwright Spec

Drop the generated .spec.ts file into a Playwright project's tests/ folder, then:

# Interactive UI mode (recommended for debugging)
npx playwright test --ui

# Headed mode — see the browser as it runs
npx playwright test flow-replay-mysite.spec.ts --headed

# Open the HTML report from the last run
npx playwright show-report

# Run with traces, screenshots, video on failure (set in playwright.config.ts)
npx playwright test --trace on --video on

First-time setup

If you don't have a Playwright project yet:

npm init playwright@latest

Then drop the spec into the generated tests/ folder.

Reading the results

After a run, four things tell you what happened:

  1. Console output — Real-time console.log from the spec (the ✅ step X, 🔄 auto-advance, 🍪 dismissed cookie lines).
  2. flow-replay-results.json — Attached to the test in the HTML report. Compact per-step OK/SKIPPED/ERROR.
  3. summary.json — Attached alongside. Aggregate totals + final URL.
  4. The trace — If trace: 'on', drag the trace.zip to trace.playwright.dev for time-travel debugging.

🆘 Troubleshooting

"CTA not found" on the first step

Cookie banner is probably intercepting the click. Run this in DevTools console on the test page to identify it:

copy([...document.querySelectorAll('[id*=cookie i],[id*=consent i],[class*=cookie i],[class*=consent i]')].map(e => e.outerHTML.slice(0,300)).join('\n---\n'))

Then share the output with whoever maintains the tool to add a selector.

"URL gate failed" after a successful click

Production probably navigated to a page that's not in the recording. Auto-advance handles this most of the time. If it gives up, the recording is too stale — re-record the flow.

Spec passes but step X was SKIPPED

By default FAIL_ON_SKIPPED = true so this should fail the test. If you want lenient mode (pass even with skips), edit the constants at the top of the generated .spec.ts.

Bookmarklet doesn't appear

Some sites have a strict Content Security Policy that blocks bookmarklets. Workaround: open DevTools console and paste the bookmarklet's code (without the javascript: prefix) directly.
