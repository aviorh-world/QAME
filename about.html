01 / How I Think
QA Philosophy
🔬
Root Cause First
Every bug I investigate, I isolate the root cause before deciding on the fix. Understanding why it happened matters more than just reproducing it.
🛡️
Prevention Over Detection
A bug found in production is a failure of the process. I focus on building coverage that catches regressions before they reach users.
🧩
System Thinking
I don't test layers in isolation. I understand how the backend, API, UI, and third-party services interact — and where they break.
📝
Knowledge Sharing
I write detailed documentation of test procedures, integration flows, and findings — so the whole team can benefit, not just me.
🤖
Automation Mindset
Every repeatable manual test is a candidate for automation. I run automated test suites via Cloudbeat, analyze results, and identify failures. I'm also actively developing Playwright skills to contribute to scripted automation.
🚀
AI-Augmented Testing
I leverage AI tools to shorten investigation time, generate test ideas, and build smarter test coverage — including tools I build myself.

Technical Context

My coding experience is recent and still developing. I work with a C# / .NET backend environment — I read and understand the codebase, work in Visual Studio, analyze API responses, trace logs, and write SQL queries for data validation.

I also write JavaScript helpers and bookmarklets to automate repetitive testing tasks, and I'm actively learning Playwright to build more structured automation. The UI Diff tool you see in this portfolio is an example of what I've built.

Automation Platform

Cloudbeat is a test management platform with a built-in test runner — similar to TestRail, but with the ability to execute tests directly from the platform. My role is to run the test suites, monitor execution results, and analyze failures — identifying root causes, regressions, and areas that need follow-up investigation.

My role is test design and failure analysis, not scripting the automation engine itself. This is how I describe it honestly: I'm automation-oriented and execution-experienced, and I'm developing my code-writing skills in parallel.

Career Direction

Yes — it's a deliberate goal, not just a trend I'm following. I approach every manual test with the question: "Could this be automated?" When I find a root cause and a repeatable scenario, my first instinct is to cover it with automation to prevent future regression.

I'm actively building Playwright skills and have already explored fixture architecture and integration test design. The transition from execution-focused QA to automation engineering is where I'm heading.

Investigation Process

I investigate systematically: logs first (Coralogix, Stackify, Kibana), then API behavior via Postman/Swagger/Fiddler, then data validation via SQL queries, then environment comparison (DEV vs QA vs UAT vs PROD).

My goal is always to isolate the root cause — not just reproduce the issue. I want to understand whether it's a Jifiti-side bug, a third-party lender issue, a configuration error, or a data problem. That distinction drives the fix and the regression coverage.

Bug Categories

I work across all layers: logic bugs (wrong business rules), data inconsistencies (wrong values, missing records), API and integration failures (wrong payloads, HTTP errors, auth issues), configuration problems (wrong URLs, missing feature flags), and regressions (things that used to work).

I'm not a UI-only tester. At Jifiti I regularly find issues that originate deep in the backend — a job that silently fails, a field that doesn't get saved, an edge case in the lender API response handling.

AI & Tooling

I actively use AI to accelerate investigation, generate test ideas, write documentation faster, and build tools. I created a NotebookLM-based knowledge session for internal QA onboarding, and built a UI comparison tool that runs deterministic checks between Figma designs and live pages.

Beyond using AI tools, I think about how to apply AI to real testing problems — which is what led me to build the UI Diff tool. The goal is always the same: reduce manual effort and increase accuracy.

Collaboration

I try to speak the developer's language. When I report a bug, I include: exact reproduction steps, relevant log lines with timestamps, API request/response payloads, SQL query results that confirm the data state, and a hypothesis on root cause.

My goal is to reduce back-and-forth. A developer should be able to read my bug report and go straight to the fix — not spend 30 minutes trying to reproduce it.

Knowledge Sharing

Yes — actively. At Jifiti I wrote multiple Jifipedia (Confluence) articles that became the standard reference for testing procedures across the R&D team: step-by-step guides for adding approvers in the John Deere system, FTP/Fiserv validation procedures, branding system documentation, and more.

I also created a structured E2E testing guide for the QA team.

Differentiator

The combination of deep system understanding + analytical investigation + AI-driven productivity. I don't just find bugs — I understand why they happen, document them clearly, and think about how to prevent them from returning.

I also build tools to solve real problems, which puts me in a different category from a traditional manual QA engineer. The UI Diff tool is one example — built from scratch to solve a specific gap in our visual testing workflow.

Career Goals

I'm aiming to grow into an automation-heavy QA or SDET role, where I can combine my backend understanding, system-level thinking, and growing Playwright/JavaScript skills to build scalable, long-term test infrastructure.

The goal isn't just to run more tests — it's to build a quality system that scales with the product and catches the right things before they reach users.
